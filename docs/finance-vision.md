## معالجة الفجوات التقنية (Mapping Layer)

### توحيد مسميات الحقول

```
┌─────────────────────┬──────────────────────┬─────────────────────────────────────┐
│ التسمية في v3.3      │ التسمية في الكود      │ آلية الربط                          │
├─────────────────────┼──────────────────────┼─────────────────────────────────────┤
│ monthlyRevenue       │ total_revenue        │ شهري مباشر (تجنب تضارب السنوي)     │
│ monthlyExpenses      │ cogs + admin_exp     │ تجميع البندين = المصروف الكلي       │
│ cashBalance          │ current_assets       │ current_assets - inventory          │
│                      │                      │ (إذا لا مخزون = كامل الأصول)       │
│ payables             │ current_liabilities  │ شمول كافة الالتزامات قصيرة الأجل    │
│ receivables          │ —                    │ حقل جديد (غير موجود بعد في الكود)  │
└─────────────────────┴──────────────────────┴─────────────────────────────────────┘
```

### معالجة المخزون (Inventory Logic)

```
المرحلة 0:  حقل اختياري
            إذا تُرِك فارغاً → كامل current_assets = سيولة جاهزة

المرحلة 4:  Small + Medium يُشجَّعون على إدخال المخزون بدقة
            ← لفصل "السيولة النقدية" عن "الأصول البطيئة"
```

```javascript
// Mapping Layer — الكود المرجعي
const mappingLayer = {
    fromCode: (codeData) => ({
        monthlyRevenue: codeData.total_revenue / 12,    // سنوي → شهري
        monthlyExpenses: (codeData.cogs + codeData.admin_expenses) / 12,
        cashBalance: codeData.current_assets - (codeData.inventory || 0),
        receivables: codeData.receivables || 0,
        payables: codeData.current_liabilities,
    }),
    
    toCode: (v33Data) => ({
        total_revenue: v33Data.monthlyRevenue * 12,     // شهري → سنوي
        cogs: v33Data.monthlyExpenses * 0.6 * 12,       // تقدير 60% تكلفة مبيعات
        admin_expenses: v33Data.monthlyExpenses * 0.4 * 12, // 40% إدارية
        current_assets: v33Data.cashBalance + (v33Data.inventory || 0),
        current_liabilities: v33Data.payables,
    }),
};
```

## ميزات UX الذكية (Smart Layers)

### 1. شريط المرجعية المالي (Reference Bar)
```
شريط علوي دائم في البوابة العميقة
يعرض: إيرادك: 45,000 ﷼ | مصروفك: 32,000 ﷼ | ربحك: 13,000 ﷼
الهدف: تذكير العميل بالأساس الذي يبني عليه تفاصيله
التحويل: الـ summary-col الحالي → شريط علوي ثابت
```

### 2. وسوم المصدر (Source Tags)
```
بجانب كل حقل مستورد: وسم "مستورد من الفحص الأولي 🔗"
CSS Tooltips / Badges صغيرة
الهدف: رفع الشفافية والثقة
```

### 3. التزامن العكسي (Back-Sync)
```
زر "تعديل الأرقام الكبرى" → يفتح الباب الرئيسي
عند التعديل: كل حسابات البوابة العميقة تتحدث تلقائياً
⚠️ يحافظ على التفاصيل الدقيقة (رواتب، إيجار..) المُدخلة سابقاً
تقنياً: State Management — cascading update
يُبنى آخر شي (الأعقد)
```

### 4. عداد المتبقي (Unallocated Counter) ⭐
```
في قسم تفصيل المصاريف:
العداد = المصروف المرجعي - مجموع البنود المُفصّلة

مثال:
المصروف المرجعي: 32,000 ﷼
├── إيجار: 8,000
├── رواتب: 15,000
├── بضاعة: 5,000
└── المتبقي: 4,000 ﷼ ← "أين تذهب هذه الفلوس؟"

إذا أصبح سالباً: "اكتشفت مصاريف خفية! مصاريفك الحقيقية أكثر مما ظننت"
← يكشف "الثقوب السوداء" — 15-30% من المصاريف المنسية
⭐ ميزة تنافسية — لم تُرَ في أي منصة سعودية
```

## أنماط الاكتشاف — بالحجم

### MICRO (1-9 موظفين)
```
🔴 "المالك يفعل كل شي"     → employees ≤ 5 + financialSystems < 3
🔴 "خلط الشخصي بالتجاري"    → employees ≤ 5 + لا محاسب
🟡 "عميل واحد = كل الإيراد"  → client_concentration > 50%
🟢 "الهامش يكفي للبقاء"     → profitMargin > 20%
```

### SMALL (10-49 موظف)
```
🔴 "نمو بدون بنية"          → revenue > 50K + financialSystems < 3
🔴 "فجوة تمويل التوسع"      → investmentNeeded > monthlyRevenue × 3
🟡 "تكاليف ثابتة عالية"     → expenses/revenue > 75%
🟡 "إيرادات غير مستقرة"     → revenue_pattern = موسمية
🟢 "تمويل ذاتي ممكن"       → (profit × 12) >= investmentNeeded
```

### MEDIUM (50+ موظف) — يُصمَّم لاحقاً
```
🔴 "تحصيل آجل خطير"        → receivables > revenue × 2
🔴 "اعتماد على خط واحد"     → منتج/خدمة واحدة > 70%
🟡 "عدم كفاءة تشغيلية"     → expense ratio > 80%
```

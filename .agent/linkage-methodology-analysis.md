### المرحلة الأولى: الحلقة المغلقة (Closed Loop Design)
**التغطية: 🟡 45% | النوع: منطق + بنية تحتية**

#### الخطوة 1.1: دورة البيانات

| المتطلب | الحالة في النظام | التفاصيل |
|---------|-----------------|---------|
| إدخال البيانات | ✅ موجود | `KPIEntry` — إدخال دوري مع فترة وحالة |
| المعالجة | ✅ جزئي | Alert Engine يحسب النسب ويقارن بالعتبات |
| التحليل | ✅ جزئي | Health Index يحسب مؤشر مركب |
| القرار | ✅ جزئي | `StrategicReview.decision` (CONTINUE/ADJUST/PIVOT) |
| التنفيذ | ❌ ناقص | لا يوجد ربط تلقائي: قرار → تعديل خطة |
| التغذية العكسية | 🟡 جزئي | Alert Engine يرسل تنبيهات، لكن لا يُعدّل الخطة تلقائياً |

**🔍 الفجوة الرئيسية:**
- الحلقة **مفتوحة** حالياً — البيانات تدخل ← تُحلل ← تُعرض، لكنها **لا تعود لتعديل الخطة تلقائياً**.
- المطلوب: عند اكتشاف انحراف كبير → تحديث أوتوماتيكي لحالة الهدف/المبادرة → إعادة حساب الأولويات.

#### الخطوة 1.2: آلية الاستشعار (Sensing)

| الحساس المطلوب | الحالة | التقنية الحالية |
|---------------|--------|----------------|
| مؤشرات أداء يومية | ✅ موجود | `KPIEntry` بدورية `DAILY/WEEKLY/MONTHLY` |
| الإنفاق الفعلي | ❌ غير موجود | `StrategicInitiative.budget` ثابت — لا يوجد `actualSpend` |
| رضا العملاء | ❌ غير موجود | لا يوجد نموذج استبيانات |
| دوران الموظفين | ❌ غير موجود | لا يوجد نموذج HR |

#### الخطوة 1.3: آلية الإجراء التصحيحي (Acting)

| القاعدة | الحالة | التنفيذ الحالي |
|---------|--------|---------------|
| انحراف > 15% → تنبيه | ✅ يعمل | Alert Engine: `warningThreshold` |
| انحراف > 25% → اجتماع طارئ | 🟡 جزئي | Alert Engine يرسل CRITICAL لكن لا يُنشئ اجتماع |
| نشاط مكتمل → تحديث مشروع | ❌ غير موجود | لا يوجد نموذج "أنشطة" مستقل |
| مشروع متأخر → تحديث مخاطر | ❌ غير موجود | `INITIATIVE_DELAYED` alert يُنشأ لكن لا يُحدّث `StrategicRisk` |

**📊 الحكم:**
- الجزء الأكبر من البنية التحتية **موجود** (Alert Engine + Health Index)
- المفقود هو **الإجراءات التصحيحية التلقائية** (Auto-Acting)
- يحتاج: Event-Action Matrix (جدول يربط كل حدث بفعل تلقائي)

**⏱️ الجهد:** 15-20 ساعة  
**🎯 التوصية:** ✅ قابل للتنفيذ — يبني على البنية الموجودة

### المرحلة الثانية: ربط SWOT بـ TOWS
**التغطية: 🟠 25% | النوع: منطق متقدم**

#### ما هو موجود:

| المكون | الحالة | الموقع في النظام |
|--------|--------|-----------------|
| نقاط SWOT | ✅ موجود | `AnalysisPoint` (STRENGTH, WEAKNESS, OPPORTUNITY, THREAT) |
| عرض SWOT Matrix | ✅ موجود | `GET /api/analysis/swot/:assessmentId` |
| ربط SWOT بتوصيات | ✅ جزئي | `alert-engine.js: /recommendations` يربط KPIs ضعيفة بـ SWOT |

#### ما هو مفقود:

| المكون المطلوب | الحالة | التحليل |
|---------------|--------|---------|
| **ترميز SWOT** (S1, W2, O3...) | ❌ غير موجود | `AnalysisPoint` لا يحتوي حقل `code` — الترميز يدوي حالياً |
| **مصفوفة TOWS** (SO, ST, WO, WT) | ❌ غير موجود | لا يوجد نموذج يربط عنصرين SWOT معاً لتوليد استراتيجية |
| **ربط SWOT → مبادرة** | ❌ غير مباشر | `StrategicInitiative` لا يحتوي `swotPointId` |
| **محرك استنتاج** (Inference) | ❌ غير موجود | لا يوجد منطق يولّد استراتيجيات من تقاطعات SWOT |
| **تقييم الاستراتيجيات** (جدوى/تكلفة/أثر) | 🟡 جزئي | `StrategicChoice` يحتوي `feasibility` و `marketAttractiveness` |

**🔍 الفجوة الرئيسية:**
```
الحالي:    SWOT (مسطح) -----> Alert Engine يستخدمها في التوصيات
المطلوب:   SWOT (مُرمّز) ---> TOWS Matrix ---> استراتيجيات مقترحة ---> مبادرات/مشاريع
```

**📊 الحكم:**
- هذه هي **الحلقة الأضعف** في النظام — SWOT موجود لكنه "معلق في الهواء"
- يحتاج: نموذج `TOWSStrategy` جديد + حقل `code` في `AnalysisPoint` + ربط مع `StrategicChoice/Initiative`
- المنهجية الحالية في `alert-engine.js` تربط KPIs بـ SWOT وهذا **بداية جيدة** يمكن البناء عليها

**⏱️ الجهد:** 20-30 ساعة  
**🎯 التوصية:** ✅ قابل للتنفيذ — عائد عالي على الاستثمار

### المرحلة الثالثة: الربط السببي BSC (Causal Linkage)
**التغطية: 🟡 35% | النوع: بنية بيانات + منطق معقد**

#### ما هو موجود:

| المكون | الحالة | الموقع |
|--------|--------|--------|
| أهداف بتصنيف BSC | ✅ موجود | `StrategicObjective.perspective` (FINANCIAL, CUSTOMER, INTERNAL_PROCESS, LEARNING_GROWTH) |
| KPIs بتصنيف BSC | ✅ موجود (جديد) | `KPI.bscPerspective` — أُضيف اليوم |
| KPIs Leading/Lagging | ✅ موجود (جديد) | `KPI.kpiType` — أُضيف اليوم |
| هرمية الأهداف | ✅ موجود | `StrategicObjective.parentId` (OKR hierarchy) |
| KPIs مربوطة بأهداف | ✅ موجود | `KPI.objectiveId` |

#### ما هو مفقود:

| المكون المطلوب | الحالة | التحليل |
|---------------|--------|---------|
| **علاقات سببية** بين الأهداف | ❌ غير موجود | `parentId` = هرمية، لكن **ليس سببية**. العلاقة السببية مختلفة: "التدريب ← تحسين عمليات ← رضا عملاء ← إيرادات" |
| **نوع العلاقة** (يؤدي إلى / يساهم في / يُمكّن) | ❌ غير موجود | لا يوجد حقل relationship type بين الأهداف |
| **قوة العلاقة** (1-3) | ❌ غير موجود | العلاقة الحالية ثنائية (parent-child) بدون وزن |
| **خريطة استراتيجية** (Strategy Map) | ❌ غير موجود | لا يوجد عرض مرئي يوضح التسلسل السببي بين المستويات |
| **محرك التحليل الجذري** (Root Cause) | 🟡 بدائي | Alert Engine يكتشف مشاكل لكن لا يتتبع السبب الجذري |

**🔍 الفجوة الرئيسية:**
```
الحالي:    Objective ← (parent) ← Objective    (هرمية بسيطة)
المطلوب:   Objective ← (causes, strength:3) ← Objective ← (enables, strength:2) ← Objective
                                                    ↑
                                           Root Cause Analysis Engine
```

**المقترح التقني:**
```prisma
// نموذج جديد مطلوب:
model CausalLink {
  id              String  @id @default(cuid())
  sourceObjectiveId String  // الهدف المصدر (السبب)
  targetObjectiveId String  // الهدف الهدف (النتيجة)
  relationshipType  String  // LEADS_TO, CONTRIBUTES_TO, ENABLES
  strength          Int     // 1=ضعيف, 2=متوسط, 3=قوي
  notes             String?
}
```

**📊 الحكم:**
- البنية الأساسية (BSC perspectives + hierarchy) **موجودة وقوية**
- المفقود هو **طبقة العلاقات السببية** — وهي الطبقة الأعلى قيمة في المنهجية
- يحتاج: نموذج `CausalLink` + واجهة Strategy Map مرئية + محرك Root Cause
- **صعوبة التنفيذ: متوسطة-عالية** (الـ Schema بسيط، لكن الواجهة والمنطق معقدان)

**⏱️ الجهد:** 25-40 ساعة  
**🎯 التوصية:** ⚠️ عائد عالي جداً — لكن يحتاج تخطيط دقيق

### المرحلة الرابعة: ربط الموارد البشرية
**التغطية: ⬜ 0% | النوع: بنية تحتية جديدة كلياً**

| المكون المطلوب | الحالة | التحليل |
|---------------|--------|---------|
| نموذج الوظائف/المناصب | ❌ غير موجود | `Member` يحدد الدور (OWNER/ADMIN) لكن ليس منصب وظيفي |
| تصنيف الأهمية الاستراتيجية | ❌ غير موجود | لا يوجد تقييم 1-5 |
| صعوبة الإحلال | ❌ غير موجود | لا يوجد |
| خطة الإحلال | ❌ غير موجود | لا يوجد |
| ربط الوظائف بالأهداف | ❌ غير موجود | `StrategicObjective.ownerId` موجود لكنه فقط FK |
| تنبيه استقالة | ❌ غير موجود | لا يوجد نموذج HR events |

**📊 الحكم:**
- هذه المرحلة **خارج نطاق النظام الحالي بالكامل**
- النظام مصمم كـ **Strategic Management** وليس **HRM**
- تنفيذها يتطلب بناء **وحدة HR كاملة** أو **تكامل مع نظام HR خارجي**
- الأقرب: حقل `ownerId` في الهدف يوفر ربط بدائي

**⏱️ الجهد:** 40-60 ساعة (وحدة جديدة)  
**🎯 التوصية:** ⏸️ مؤجلة — يُفضل التكامل مع نظام HR بدلاً من بنائه

### المرحلة الخامسة: التشغيل البيني للبيانات
**التغطية: 🟡 30% | النوع: بنية تحتية + معايير**

#### ما هو موجود:

| المكون | الحالة | الموقع |
|--------|--------|--------|
| أكواد القطاعات | ✅ موجود | `Sector.code` (unique) |
| أكواد الأنشطة | ✅ موجود | `Industry.code` (unique) |
| أكواد الكيانات | ✅ ضمني | `Entity.id` (cuid) |
| نظام تكامل خارجي | ✅ بنية أولية | `Integration` model (provider, webhookUrl, apiKey) |
| API موحد | ✅ موجود | Express REST APIs |

#### ما هو مفقود:

| المكون المطلوب | الحالة | التحليل |
|---------------|--------|---------|
| معايير ISIC | ❌ غير مستخدم | `Industry.code` موجود لكن لا يتبع ISIC Rev.4 |
| قواميس بيانات | ❌ غير موجود | لا يوجد Data Dictionary رسمي |
| جداول مرجعية | 🟡 جزئي | `Sector`, `Industry`, `EntityType` — لكن بدون معايير دولية |
| APIs للتكامل الفعلي | ❌ غير مفعّل | `Integration` model موجود لكن sync.js يعمل داخلي فقط |
| Webhooks | 🟡 بنية فقط | `Integration.webhookUrl` موجود لكن لا يوجد webhook dispatcher |

**📊 الحكم:**
- البنية الأساسية **موجودة وجاهزة للتوسيع**
- المفقود الرئيسي: اعتماد **معايير دولية** (ISIC, ISO) + **Webhook Dispatcher** + **Data Dictionary**
- هذه مرحلة **معيارية** أكثر من كونها برمجية

**⏱️ الجهد:** 30-50 ساعة  
**🎯 التوصية:** 🟡 متوسطة — مهمة للتوسع لكن ليست عاجلة

### المرحلة السادسة: الجهاز العصبي الاستراتيجي
**التغطية: 🟠 20% | النوع: معمارية متقدمة**

#### تحليل العقد الخمس:

| العقدة | الحالة في النظام | النضج |
|--------|-----------------|--------|
| **عقدة الإدخال** (الأنشطة/المهام) | ❌ غير موجود | لا يوجد نموذج Task/Activity مستقل — فقط `Initiative` |
| **عقدة المعالجة** (المشاريع/البرامج) | ✅ موجود | `StrategicInitiative` (progress, status, budget) |
| **عقدة التحليل** (المؤشرات/المقاييس) | ✅ قوي | `KPI` + `KPIEntry` + `Health Index` (أفضل عقدة) |
| **عقدة القرار** (الأهداف) | ✅ موجود | `StrategicObjective` + `StrategicReview.decision` |
| **عقدة الموارد** (الميزانيات/الأشخاص) | 🟡 ضعيف | `Initiative.budget` فقط — لا HR |

#### تحليل المشابك (Synapses):

| المشبك المطلوب | الحالة | التنفيذ |
|---------------|--------|---------|
| نشاط → مشروع | ❌ غير موجود | لا يوجد نموذج Task |
| مشروع → مؤشر | 🟡 ضعيف | `Initiative.kpiId` موجود لكن نادراً ما يُستخدم |
| مؤشر → هدف | ✅ قوي | `KPI.objectiveId` |
| هدف → ميزانية | ❌ غير مباشر | لا يوجد ربط مباشر بين Objective و Budget |
| **انتشار تلقائي** | 🟡 جزئي | Alert Engine يكتشف لكنه لا ينشر التحديث للعقد المرتبطة |

**📊 الحكم:**
- النظام يملك **3 من 5 عقد** بدرجة نضج مقبولة
- المشبك الأقوى: `KPI → Objective` — والأضعف: `Task → Initiative → KPI`
- **الانتشار التلقائي** (Cascading Updates) هو الحلقة المفقودة الأهم

**⏱️ الجهد:** 50-80 ساعة  
**🎯 التوصية:** ⚠️ هذه المرحلة هي **الرؤية النهائية** — تُبنى تدريجياً

## ⚖️ تقييم الجدوى الكلي

### نقاط القوة في النظام الحالي:
1. **✅ Schema ناضج** — يغطي 80% من النماذج المطلوبة
2. **✅ BSC مُدمج** — perspectives + leading/lagging + hierarchy
3. **✅ Alert Engine قوي** — يكتشف 5 أنواع من المشاكل
4. **✅ Health Index** — معادلة مركبة 4 أبعاد
5. **✅ SWOT موجود** — AnalysisPoint + Assessment
6. **✅ Review System** — decision engine (CONTINUE/ADJUST/PIVOT)

### نقاط الضعف الهيكلية:
1. **❌ لا يوجد Causal Links** — العلاقات هرمية فقط وليست سببية
2. **❌ لا يوجد TOWS Matrix** — SWOT لا يُولّد استراتيجيات
3. **❌ لا يوجد Task/Activity** — المستوى التشغيلي مفقود
4. **❌ لا يوجد Auto-Acting** — الحلقة مفتوحة (detection بدون correction)
5. **❌ لا يوجد HR Module** — ربط الموارد البشرية غائب كلياً
6. **❌ لا يوجد Test Suite** — لا اختبارات آلية

## 🎯 الخلاصة النهائية

| السؤال | الإجابة |
|--------|---------|
| **هل المنهجية قابلة للتطبيق؟** | ✅ نعم — لكن **تدريجياً** وليس دفعة واحدة |
| **ما الجزء الأسهل؟** | المراحل 0, 7, 8 (وثائقية + اختبار) |
| **ما الجزء الأعلى قيمة؟** | المراحل 2, 3 (TOWS + BSC Causal Linkage) |
| **ما الجزء غير القابل للتنفيذ حالياً؟** | المرحلة 4 (HR) — يحتاج نظام HR |
| **الجهد الكلي؟** | ~200-350 ساعة (8-14 أسبوع عمل) |
| **أين نبدأ؟** | المرحلة 1 (إغلاق الحلقة) ← المرحلة 2 (TOWS) ← المرحلة 3 (BSC Causal) |
